# Demo Fullstack -- Task Board

A three-tier Kanban task board demonstrating how to deploy a production-quality
application on the SRE Kubernetes platform. Use it as a reference for building
your own apps with the `sre-web-app` and `sre-api-service` Helm templates.

## Architecture

```
Browser --> taskboard.apps.sre.example.com (HTTPS)
             |
        Istio Gateway + OAuth2 Proxy (SSO)
             |
     VirtualService path routing
             |
    /api/*   |    /*
      |      |     |
  +--------+ | +----------+
  |Backend | | |Frontend  |
  |Go :9090| | |nginx:8080|
  +--------+ | +----------+
      |       
  +--------+
  |Postgres|
  | :5432  |
  +--------+
```

The Istio VirtualService (`api-virtualservice.yaml`) routes `/api/*` to the Go
backend and everything else to the nginx frontend. OAuth2 Proxy handles SSO for
both paths through `/oauth2/` routes on the same hostname.

## For Developers

If you want to deploy an app like this, you don't need anything in the `k8s/` directory.
See the developer-friendly version: **[Example 07 - Fullstack App](../../tools/developer-kit/examples/07-fullstack-app/)**

That example shows just the `bundle.yaml` you would create. The platform generates
all the Kubernetes manifests (HelmReleases, VirtualServices, NetworkPolicies, etc.)
automatically from your bundle.

## Components

### Frontend (React + nginx)

- React 18, TypeScript, Tailwind CSS, Vite
- Drag-and-drop Kanban columns: Todo, In Progress, Done
- Served by `nginxinc/nginx-unprivileged:1.27-alpine` on port 8080
- Deployed via `sre-web-app` HelmRelease with Istio VirtualService for ingress
- Image: `harbor.apps.sre.example.com/platform/demo-frontend:v1.0.6`

### Backend (Go REST API)

- Go 1.22 stdlib with `net/http` enhanced routing (no frameworks)
- CRUD on tasks with PostgreSQL via `lib/pq`
- Prometheus metrics at `/metrics` (`demo_http_requests_total`, `demo_http_request_duration_seconds`)
- Health: `/healthz` (liveness), `/readyz` (readiness, checks DB)
- Structured JSON logging to stdout
- Graceful shutdown on SIGTERM
- Deployed via `sre-api-service` HelmRelease on port 9090
- Image: `harbor.apps.sre.example.com/platform/demo-backend:v1.0.0`

### Database (PostgreSQL)

- PostgreSQL 16.4-alpine StatefulSet, 1 replica
- 1Gi PVC on `local-path` StorageClass
- Init SQL seeds 5 example tasks via ConfigMap
- Credentials from a Kubernetes Secret (dev) or ExternalSecret (prod)
- Runs as non-root (UID 999), drops all capabilities

## SRE Platform Integration

### Istio

- **Path-based routing**: `api-virtualservice.yaml` sends `/api/*` to the
  backend and `/oauth2/*` to OAuth2 Proxy, all on the same hostname. The
  frontend VirtualService (generated by sre-web-app) handles `/*`.
- **mTLS**: Inherited from cluster-wide STRICT PeerAuthentication.
- **AuthorizationPolicy**: Backend allows traffic from `team-demo` and
  `istio-system` namespaces only.

### NetworkPolicies

- **Database**: Ingress only from pods labeled `demo-backend`, egress only to
  kube-system DNS.
- **Backend**: Ingress from frontend and istio-system, egress to database port
  5432 (via sre-api-service chart).
- **Frontend**: Ingress from istio-system gateway, egress to backend port 9090
  (via sre-web-app chart).

### Monitoring

Backend ServiceMonitor scrapes `/metrics` every 15s. Exposed metrics:
- `demo_http_requests_total{method, path, status_code}` -- request counter
- `demo_http_request_duration_seconds{method, path}` -- latency histogram

### Secrets Management

Dev uses a plain Kubernetes Secret (`database-secret.yaml`). For production,
replace it with the ExternalSecret in `externalsecret.yaml`, which syncs
credentials from OpenBao via the ClusterSecretStore:

```bash
vault kv put sre/data/team-demo/demo-db \
  username=taskboard password=<secure-password> database=taskboard
```

Then swap `database-secret.yaml` for `externalsecret.yaml` in
`k8s/kustomization.yaml`.

### GitOps (Flux)

The Flux Kustomization lives at `platform/addons/components/demo-fullstack.yaml`.
It uses `postBuild.substituteFrom` to inject `SRE_DOMAIN` from the
`sre-domain-config` ConfigMap. Dependencies: `sre-istio`, `sre-tenants`.

## Building Your Own Fullstack App

**For developers:** You don't need to touch HelmReleases, VirtualServices, or Flux configs.
Create a `bundle.yaml` describing your app and its components, then submit through the
DSOP Wizard. See [Example 07](../../tools/developer-kit/examples/07-fullstack-app/) for
exactly what you need to create.

Your `bundle.yaml` just needs:
1. Your frontend image under `spec.app`
2. Your backend image under `spec.components`
3. `services.database.enabled: true` if you need PostgreSQL
4. Your health check endpoints under `probes`

The platform generates all the Kubernetes manifests automatically.

## API Reference

All endpoints are under `/api/` and return JSON.

| Method | Path              | Description         | Request Body                                  |
|--------|-------------------|---------------------|-----------------------------------------------|
| GET    | /api/tasks        | List all tasks      | --                                            |
| POST   | /api/tasks        | Create a task       | `{"title": "...", "description": "...", "status": "todo"}` |
| PUT    | /api/tasks/{id}   | Update a task       | `{"title": "...", "status": "in-progress"}`   |
| DELETE | /api/tasks/{id}   | Delete a task       | --                                            |

Valid status values: `todo`, `in-progress`, `done`. Title is required on create.
PUT accepts partial updates (only send the fields you want to change).

Health and observability (not behind `/api/`):

| Path     | Purpose              |
|----------|----------------------|
| /healthz | Liveness probe       |
| /readyz  | Readiness probe (DB) |
| /metrics | Prometheus metrics   |

## Troubleshooting

**Pods stuck in CrashLoopBackOff (backend)**
- Check database connectivity: `kubectl logs -n team-demo deploy/demo-backend-demo-backend`
- Verify the `DATABASE_URL` env var and that the `demo-db` pod is running.

**502 Bad Gateway on /api/ routes**
- Verify the backend service exists: `kubectl get svc -n team-demo`
- Check VirtualService destination matches the actual service name:
  `kubectl get vs -n team-demo demo-backend-api -o yaml`

**Frontend loads but API calls fail**
- Open browser devtools Network tab. If `/api/tasks` returns a CORS error or
  503, the backend pod may not be ready.
- Check Istio sidecar injection: `kubectl get pod -n team-demo -o jsonpath='{.items[*].spec.containers[*].name}'` should show `istio-proxy` sidecars.

**Database data lost after restart**
- Verify the PVC exists: `kubectl get pvc -n team-demo`
- StatefulSet PVCs survive pod restarts. Data is only lost if the PVC is deleted.

## Lesson Learned: Istio Path Routing

Do NOT proxy between services using nginx inside pods. A common mistake is to add
a `proxy_pass` block in nginx.conf to forward `/api/*` to the backend. This
fights against the Istio sidecar's Envoy proxy, causing `BlackHoleCluster`
routing failures because Envoy intercepts all outbound traffic.

The correct pattern is Istio VirtualService path-based routing at the gateway
level. The gateway sees the full URL path and routes to the right service before
traffic enters any pod. This is visible in `api-virtualservice.yaml`: it matches
`/api/` and sends it to the backend service, while the frontend VirtualService
(from the sre-web-app chart) handles everything else. Each pod only serves its
own content -- no cross-service proxying.
