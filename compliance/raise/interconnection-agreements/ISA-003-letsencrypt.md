# Interconnection Security Agreement (ISA)

## ISA-003: Let's Encrypt (Certificate Issuance)

| Field | Value |
|-------|-------|
| **ISA ID** | ISA-003 |
| **Connected System** | Let's Encrypt ACME CA (letsencrypt.org) |
| **System Owner** | Internet Security Research Group (ISRG) |
| **Connection Purpose** | Automated TLS certificate issuance and renewal via ACME protocol |
| **Classification** | Unclassified |
| **Effective Date** | 2025-06-15 |
| **Review Date** | 2026-06-15 |
| **Status** | Active (dev/staging); Internal CA used in production |

---

## 1. Connection Description

The SRE Platform uses cert-manager to automate TLS certificate lifecycle management. In development and staging environments, cert-manager connects to Let's Encrypt via the ACME protocol to obtain publicly trusted TLS certificates for ingress endpoints.

In production and government deployments, an internal CA (or DoD PKI) is used instead. Let's Encrypt is used only for non-production environments where publicly trusted certificates simplify testing.

## 2. Connection Details

| Parameter | Value |
|-----------|-------|
| **Protocol** | HTTPS (ACME v2) |
| **Port** | 443 |
| **Direction** | Outbound (cert-manager to Let's Encrypt) |
| **Endpoint (staging)** | acme-staging-v02.api.letsencrypt.org |
| **Endpoint (production)** | acme-v02.api.letsencrypt.org |
| **Authentication** | ACME account key (auto-generated by cert-manager) |
| **Challenge Type** | HTTP-01 (via Istio gateway) or DNS-01 (via cloud DNS provider) |
| **Rate Limits** | 50 certificates/domain/week (production) |

## 3. Data Transmitted

| Data Type | Direction | Classification | Description |
|-----------|-----------|---------------|-------------|
| Certificate Signing Request (CSR) | SRE -> Let's Encrypt | Unclassified | Domain names, public key |
| Signed Certificate | Let's Encrypt -> SRE | Unclassified | X.509 TLS certificate with chain |
| ACME challenge tokens | Bidirectional | Unclassified | Domain validation tokens |
| Account metadata | SRE -> Let's Encrypt | Unclassified | Contact email, account key |

**Data NOT transmitted:**
- Private keys (generated and stored locally in Kubernetes Secrets)
- Internal service names or cluster topology
- Any CUI or classified data

## 4. Security Controls

| Control | Implementation |
|---------|---------------|
| **Key Protection** | Private keys generated in-cluster and stored as Kubernetes Secrets; never transmitted (SC-12) |
| **Certificate Rotation** | cert-manager auto-renews certificates 30 days before expiry (IA-5) |
| **Monitoring** | Prometheus alerts on certificate expiry; Grafana dashboard tracks cert health (CA-7) |
| **Fallback** | Internal CA ClusterIssuer available if Let's Encrypt is unreachable (CP-2) |
| **Scope Limitation** | Only ingress gateway certificates use Let's Encrypt; internal services use Istio mTLS (SC-8) |
| **Audit** | cert-manager events logged; certificate lifecycle tracked in Kubernetes events (AU-2) |

## 5. Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Let's Encrypt CA outage prevents cert renewal | Low | Low | 30-day renewal window; internal CA fallback |
| Let's Encrypt root certificate revocation | Very Low | Medium | Certificates replaced within renewal window; internal CA available |
| ACME account key compromise | Very Low | Medium | Key stored as K8s Secret; RBAC-restricted access |
| DNS-01 challenge exploited for domain hijack | Very Low | High | DNS provider secured with MFA; challenge records scoped |

## 6. Residual Risk

**Accepted Risk:** Let's Encrypt certificates are used only in non-production environments. Production deployments use internal CA or DoD PKI chain. The risk of Let's Encrypt operational issues affecting production is zero.

## 7. NIST Control Mapping

- **IA-5**: Automated certificate lifecycle management via cert-manager
- **SC-8**: TLS encryption for all ingress traffic
- **SC-12**: Private key generation and protection within cluster boundary
- **SC-13**: FIPS-compliant TLS cipher suites enforced at Istio gateway
